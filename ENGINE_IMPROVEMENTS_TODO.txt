ISSUE 1: Native SIGSEGV/SIGABRT crash from sherpa-onnx libc++ conflict
  Status: RESOLVED via process isolation
  Files: AndroidManifest.xml, ReaderApplication.kt, NotificationReaderService.kt,
         TtsBridge.kt, TtsCommandReceiver.kt
  Root cause: sherpa-onnx ships its own native libc++. HWUI (Android's UI
    rendering) uses the system libc++. When both run in the same process,
    the two libc++ versions clash and corrupt a mutex HWUI uses for its
    worker threads, causing SIGABRT.
  Resolution: TTS engine runs in a separate `:tts` process (android:process=":tts").
    Main process: UI only (no TTS engine loaded).
    :tts process: TTS engine only (no HWUI).
    Cross-process IPC via TtsBridge: commands as broadcasts, status as JSON file.
    SherpaEngine.warmUp() is called eagerly in the :tts process onCreate().
    NOTE: It is now SAFE to call warmUp() in NotificationReaderService.onCreate()
    because the service runs in the :tts process which has no HWUI.

ISSUE 2: Piper voice extraction slow on first launch
  Status: RESOLVED
  Resolution: Voices are now split into bundled (8 in APK assets) and
    downloadable (36 via GitHub Releases). Bundled voices load directly
    from AssetManager — no extraction. Downloaded voices are saved to
    filesDir as .onnx files. No extraction step exists anymore.
    VoiceDownloadManager handles on-demand downloads with progress tracking.

ISSUE 3: Android 13+ restricted settings for sideloaded apps
  Status: RESOLVED
  Files: HomeFragment.kt, fragment_home.xml
  Problem: Sideloaded APKs on Android 13+ have notification listener toggle
    greyed out until user enables "Allow restricted settings" from app info.
  Resolution: HomeFragment now shows contextual step-by-step instructions
    tied to permission state. On Android 13+: "Step 1: tap ⋮ → Allow
    restricted settings. Step 2: go back → enable Kyōkan." Instructions
    auto-hide once permission granted and service alive. Permission status
    polls every 2s for 30s after returning from settings to catch delayed
    propagation on some OEMs.

ISSUE 4: Crash loop on Xiaomi/MediaTek devices
  Status: RESOLVED
  File: ReaderApplication.kt
  Problem: Each crash restart triggered detectPreviousCrash() → captureLogcat()
    which spawned subprocesses during HWUI init → resource contention → crash → repeat.
  Resolution: Added rapid-crash limiter (>3 crashes in 60s skips logcat capture
    to break the loop) and 5-second delay before logcat capture to let HWUI finish.

SECTION 4 — ARCHITECTURE QUICK REFERENCE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Process architecture (two-process model):
  MAIN PROCESS (default):
    - MainActivity, HomeFragment, ProfilesFragment, AppsFragment, RulesFragment
    - ReaderApplication: crash detection, log directory, uncaught exception handler
    - NO TTS engine loaded — HWUI runs safely here

  :tts PROCESS (android:process=":tts"):
    - NotificationReaderService: notification capture, filtering, dedup, mood, synthesis
    - TtsCommandReceiver: receives broadcast commands from main process
    - SherpaEngine: Kokoro + Piper TTS engines
    - AudioPipeline: FIFO queue, background playback thread
    - VoiceCommandListener: wake word + voice commands

  Cross-process IPC (TtsBridge):
    - Commands (UI → TTS): explicit broadcasts (test speak, stop, preload, voice cmd)
    - Status (TTS → UI): JSON file (tts_status.json), atomic write (tmp + rename),
      polled every 1s by UI. Fields: ready, status, error, alive, initProgress,
      voiceCmdListening, voiceCmdWakeWord.

Pipeline flow (all in :tts process):
  1. Android OS → onNotificationPosted(sbn)
  2. NotificationReaderService: filtering, dedup, sender history, flood count
  3. SignalExtractor.extract(): keywords, emoji, intent, urgency, sentiment,
     negation, intensity scaling, sarcasm, emotional blend
  4. MoodState.decayed() → MoodUpdater.update()
  5. VoiceModulator.modulate(): time-of-day → mood → signal × sensitivity
     → blend → flood tier → clamp → ModulatedVoice
  6. VoiceTransform.process(): wording rules → commentary → fillers →
     breath → gimmicks → intonation → stutter → trailing off
  7. SherpaEngine.synthesize() or synthesizePiper(): TTS → PCM samples
  8. AudioDsp.apply(): saturation → formant smooth → breathiness noise →
     spectral tilt → jitter → vocal fry → volume → trailing off fade
  9. AudioPipeline.playPcm(): AudioTrack output with pitch adjustment

Key files:
  - SherpaEngine.kt        → TTS engine wrapper (Kokoro + Piper), init progress tracking
  - AudioPipeline.kt       → FIFO queue, background playback thread
  - AudioDsp.kt            → PCM-level effects (post-synthesis)
  - VoiceModulator.kt      → Parameter computation (pitch/speed/breath/etc.)
  - VoiceTransform.kt      → Text-level effects (pre-synthesis)
  - SignalExtractor.kt     → Notification → signal map
  - MoodState.kt           → Persistent mood (valence/arousal/stability)
  - PhonicAnalyzer.kt      → Voiced region / phrase ending detection
  - VoiceProfile.kt        → User-editable voice configuration
  - NotificationReaderService.kt → Entry point, service lifecycle (:tts process)
  - TtsBridge.kt           → Cross-process IPC (commands + status)
  - TtsCommandReceiver.kt  → Broadcast receiver for commands (:tts process)
  - ReaderApplication.kt   → Crash detection, rapid-crash limiter, log resolution

Design principles (from improvement plan Part 9 — "What Not To Change"):
  - Single pipeline thread (serial FIFO, no concurrent synthesis)
  - No LLM in signal extraction (pattern matching: honest, fast, offline)
  - Quadratic smoothing curve for perceptual mapping
  - Personality presets as first-class transparent objects
  - Gimmick text variants coherent with TTS voice timbre
  - Process isolation: TTS in :tts, UI in main — never mix HWUI + sherpa-onnx
